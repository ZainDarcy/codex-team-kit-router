# AI 执行手册

本手册定义 Codex 主线程和子代理在项目内协作时的执行流程。根入口 `AGENTS.md` 只做路由，本文件承接具体执行规则。

## 最高优先级原则

- 先判断任务意图、约束和成功标准，再决定是否实现。
- 用户要求“只规划”“只测评”“先别改代码”时，默认不改文件。
- 发现更稳的方案时，先说明取舍；如果会改变用户目标，先征求确认。
- 保持 V1 范围克制，不为未确认的未来需求提前堆复杂度。
- 不把临时验证思路沉淀成正式规则，除非用户明确确认。

## 上下文加载原则

- 常驻上下文只保留 `AGENTS.md`、硬性禁令和文档路由。
- 开始较大任务、恢复上下文或判断下一步时，读取 `Docs/01-项目/项目进度.md`。
- 准备修改前，读取本文件和与任务相关的专题文档。
- 不把 `Docs/03-团队/Agents/`、`.codex/team/` 或历史记录整篇常驻注入。
- 派发子代理前，主线程应先压缩上下文，只给子代理完成任务所需的文件、约束、验收标准和未知项。

## 接入与迁移判断

把本 kit 接入已有项目时，先完成探测，不要先复制再补丁：

- 用户提供 GitHub 地址时，先拉取到 staging 目录；禁止直接 clone/解压到目标项目根目录并长期保留。
- staging 优先放在项目根目录外；若临时放在项目内，使用 `_codex-team-kit-router-staging/` 并在完成后删除。
- 确认唯一 `docs_root`，例如 `Docs/` 或 `docs/`，并同步写入 `.codex/team-kit.toml`。
- 确认真相源策略：采用已有项目规范/项目进度、迁移到推荐路径，或创建只含链接的索引页。
- 同类真相源只能有一个；不要让根目录、文档根目录和 `01-项目/` 同时保存同义内容。
- 按用户目标选择最小模式、团队就绪模式或完整模式。
- 必须选择行业扩展包：`none`、`game-basic`、`game-full` 或 `custom`；未选择的扩展包不得复制到 `.codex/agents/`。
- 初始化前扫描 `.DS_Store`，若目标项目是 Git 仓库，建议 `.gitignore` 包含 `.DS_Store`。
- 模板仓库自身的 `Docs/01-项目/项目进度.md` 只记录模板占位/示例状态，不写入目标项目的执行历史。
- 合并和体检完成后删除 staging 目录；如果删除失败，必须在最终汇报中说明残留路径和原因。

## 执行前判断

每次准备修改前，主线程必须先完成：

- 列出预计涉及文件。
- 判断任务应走 `Quick`、`Project`、`Team`、`Team-Init`、`Review` 或 `Agent-Setup` 路由。
- 如果走 `Team-Init` 或 `Agent-Setup`，先确认 `Docs/03-团队/Agents/团队初始化.md` 已执行；若未执行，先随机起名并创建团队名册和成员档案。
- 如果 `.codex/team-kit.toml` 与实际目录不一致，先修正配置或向用户说明冲突，不继续扩大改动。
- 如果项目选择了行业扩展包，确认被选中的 agent 已从 `.codex/agent-packs/` 合并到 `.codex/agents/`，未选中的扩展包不进入团队名册。
- 团队成员显示名应匹配目标项目用户的主要语言，使用自然、好记的人名；机器字段 `name` 保持英文稳定。
- 判断是否需要子代理；没有用户明确授权时不派发。
- 判断方案是否和项目规范、当前进度、公共文件写锁冲突。
- 判断是否存在更简单、更小、更可维护的方案。
- 如果用户给出的做法和项目事实冲突，先说明冲突点，不直接执行。

## 何时必须先问用户

以下情况必须先给方案和取舍，等待用户确认后再改：

- 会改变用户原始目标、范围或验收标准。
- 涉及架构归属、系统边界、公共文件结构或团队流程规则。
- 可行方案不止一个，且会影响后续维护成本或协作方式。
- 需要新增通用封装、公共规则、hooks、自动化拦截或全局模板。
- 需要删除、覆盖或迁移用户已有内容。

## 子代理派发要求

- 派发前必须写 Delegation Card，见 `.codex/team/dispatch-protocol.md`。
- 每个子代理只接收压缩上下文包，不接收整套项目文档。
- 每个子代理只能有一个主目标和一个明确交付格式。
- 实现型子代理必须拥有互不重叠的写入范围。
- 子代理不得修改公共文件，不得启动下一层子代理。
- 子代理若无法写工作记录，应把压缩的工作记录草稿和成员档案更新草稿返回给主线程。
- 主线程等待本波子代理完成后，统一验收、整合和回写公共文件。

Team 路由硬性阶段（已初始化项目）：

1. 派发前输出 Delegation Card。
2. 启动本波子代理。
3. 等待本波全部完成，并记录所有本波子代理状态。
4. 汇总建议和冲突点。
5. 主线程确认公共文件写入方案。
6. 按 Team-lite / delta policy 落盘团队沉淀。
7. 最后统一更新公共文件、项目进度和最终汇报。

如果主线程必须在子代理未全部完成前处理紧急修复，应在工作记录中写明例外原因和风险。

Team-lite / delta policy：

- 每波至少沉淀一份工作记录（可由主线程汇总本波多名参与者），但记录里必须逐一列出参与者、任务、产出、验证和风险。
- 成员档案只在出现“新增可复用经验”时更新，不做每轮机械改写。
- 团队名册每波只做轻量更新：参与者最近任务、任务次数、在岗/停用、职责变化和可用性变化；不得整篇重写。
- 初始化信息、迁移细节、扩展包选择仅在 `Team-Init` 路由处理，不作为日常 Team 默认读取项。

## 执行后同步

- 完成后列出实际涉及文件。
- 如果实际涉及文件和预计文件不一致，说明差异原因。
- 有实现、配置、文档或团队流程进展时，同步更新 `Docs/01-项目/项目进度.md`。
- 走 `Team` 路由或启用子代理时，必须确认至少一份工作记录已经落盘，且团队名册已按参与者更新最近任务和任务次数；成员档案按 Team-lite / delta policy 增量更新。
- 工作记录必须包含 `_工作记录模板.md` 的核心标题，不能只写自由格式摘要。
- 接入/初始化任务完成时，必须确认 staging 目录未残留在项目根目录。
- 运行与改动范围匹配的验证；不能运行时说明原因和风险。
- 最终回复优先说明结果、文件、验证、未验证项和残余风险。
- 最终回复必须包含团队贡献汇报，说明谁干了什么、产出了什么、哪些验证或建议进入最终交付。

## 团队贡献汇报

每次任务执行完毕后，主线程必须给用户一个简短贡献汇报：

- `Codex 主线程`：说明完成的需求理解、路由、修改、整合、验证和公共文件回写。
- `子代理角色/昵称`：如启用子代理，逐个说明它负责的任务、产出物、验证结果和被采纳的建议。
- `未启用子代理`：如没有启用子代理，说明原因，例如任务较小、没有用户授权或关键路径更适合主线程完成。
- 保持摘要化，不粘贴子代理完整日志、长命令输出或大段文档原文，避免污染主线程上下文。

建议格式：

```text
团队贡献：
- Codex 主线程：...
- qa_reviewer / Verity：...

产出：
- ...
```

## 团队沉淀收尾

每次走 `Team` 路由或启用任何子代理，最终回复前必须完成以下检查：

1. 至少写入一份本波工作记录：`Docs/03-团队/Agents/工作记录/YYYY-MM-DD-{task_slug}.md`；多人协作时必须逐一列出参与者贡献。
2. 团队名册必须按参与者更新最近任务和任务次数；只做行级轻量更新，不做无信息增量改写。
3. 仅在新增可复用经验时，创建或更新成员档案：`Docs/03-团队/Agents/成员档案/{role}-{nickname}.md`。
4. 如果子代理是 `read-only`，主线程必须把它返回的压缩草稿写入对应文件，或单独派发 `docs_keeper` 写入。
5. 如果本轮用户要求团队流程但没有启用子代理，创建 `main-thread` 工作记录，说明未派发原因、主线程完成事项和验证情况。

团队沉淀只写压缩事实：任务、产出、文件、验证、风险、下次可复用经验。不要粘贴完整子代理对话、长命令输出或大段原文。

## 输出格式要求

开始修改前：

```text
预计涉及文件：
- ...

准备做什么：
- ...
```

完成后：

```text
实际涉及文件：
- ...

验证：
- 已运行：
- 未运行：
- 风险：

团队贡献：
- Codex 主线程：
- 子代理：

产出：
- ...
```
