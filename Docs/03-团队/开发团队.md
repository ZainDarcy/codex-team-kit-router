# 开发团队

本文件记录本项目的 AI 开发团队结构、调度规则、交付标准和验收门禁。它不是固定组织架构，Codex 主线程应根据项目需求、阶段和风险增删 `.codex/agents/*.toml`。

## 团队搭建原则

- 先小队，再扩展。
- 每个任务只有一个主责 agent。
- 每个任务可以有多个协作 agents。
- 每个任务必须有验收 agent。
- 公共文件只由 Codex 主线程修改。
- 子代理完成后必须写工作记录。
- 每次任务完成后，主线程必须汇总团队贡献：谁干了什么、产出了什么、哪些验证或建议进入最终交付。

## 默认核心小队

| Agent | 职责 | 默认模型 | Sandbox | 何时使用 |
| --- | --- | --- | --- | --- |
| `producer` | 总目标、范围、价值和最终验收 | `gpt-5.5` | read-only | 高层拆解、范围取舍、最终验收 |
| `project_manager` | 任务拆解、依赖、进度和风险 | `gpt-5.4` | read-only | 排期、状态、阻塞、席位管理 |
| `product_strategist` | 需求、用户场景、产品方案 | `gpt-5.5` | read-only | PRD、功能取舍、验收口径 |
| `tech_architect` | 架构、技术方案和风险 | `gpt-5.5` | read-only | 跨模块设计、高风险实现前评审 |
| `ux_designer` | 信息架构、流程和界面体验 | `gpt-5.4` | read-only | UI/UX、流程、可用性 |
| `implementation_engineer` | 受控范围内实现代码 | `gpt-5.3-codex` | workspace-write | 小到中等代码实现 |
| `qa_reviewer` | 验收、回归、风险和缺陷 | `gpt-5.4` | read-only | 测试、审查、复测、验收 |
| `docs_keeper` | 文档草稿、工作记录和知识沉淀 | `gpt-5.4-mini` | workspace-write | 文档整理和建议回写 |
| `researcher` | 资料、竞品、技术文档和代码映射 | `gpt-5.4-mini` | read-only | 调研、探索、证据收集 |

## 主责边界

- `producer` 负责最终取舍、版本边界和验收结论。
- `product_strategist` 负责用户目标、需求表达、非目标和验收口径。
- `project_manager` 负责执行拆解、依赖、状态、风险登记和 agent 席位管理。

## 扩展专家组

根据项目领域扩展，而不是一次性启用所有岗位。

| 领域 | 可扩展 agents |
| --- | --- |
| Web / SaaS | frontend_engineer, backend_engineer, devops_engineer, data_analyst, growth_operator |
| App / 客户端 | mobile_engineer, platform_integrator, release_manager |
| 游戏 | game_designer, gameplay_engineer, ui_artist, technical_artist, playtest_researcher |
| AI / 数据 | data_engineer, ml_engineer, evaluation_engineer, mlops_engineer |
| 内容 / 媒体 | editor, visual_designer, content_ops, review_moderator |

## 需求执行流程

1. 主线程读取需求和公共文档。
2. 主线程判断是否需要子代理。
3. 主线程按 `.codex/team/model-routing.md` 选择模型和 agents。
4. 主线程派发一波任务，并明确每个子代理的允许路径、禁止路径、交付物和日志路径。
5. 主线程等待本波全部完成，不中途打断子代理。
6. 子代理写工作记录并返回结果。
7. 主线程关闭完成的子代理线程，释放席位。
8. 验收 agent 审查产出。
9. 主线程统一修改公共文件和项目进度。
10. 主线程向用户汇报团队贡献、产出、验证和未采纳建议。

## 验收门禁

| 门禁 | 验收人 | 通过条件 |
| --- | --- | --- |
| 需求门禁 | producer / product_strategist | 目标、范围、成功标准清晰 |
| 技术门禁 | tech_architect | 实现路径、风险、测试方式明确 |
| 执行门禁 | implementation_engineer | 只改允许范围，验证改动 |
| QA 门禁 | qa_reviewer | 问题、复测、未测项清晰 |
| 进度门禁 | project_manager | 状态、风险、下一步已更新 |

## 工作记录

子代理每次任务结束后，在 `Docs/03-团队/Agents/工作记录/` 创建工作记录。文件名建议：

```text
YYYY-MM-DD-角色-昵称-任务名.md
```

如果子代理无法写文件，应把完整工作记录内容返回给主线程，由主线程写入。

## 任务完成汇报

最终汇报应覆盖：

- 主线程完成的工作。
- 每个子代理完成的工作和产出。
- 被采纳的子代理建议。
- 未采纳建议及原因。
- 本轮未启用子代理时的原因。
