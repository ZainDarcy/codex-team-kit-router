# 团队初始化

本文件定义模板接入目标项目后的团队初始化动作。模板仓库不固定团队成员人名；用户把 GitHub 地址发给目标项目里的 AI 后，由 AI 在目标项目初始化时随机生成成员显示名。

## 何时必须执行

- 第一次在目标项目使用本模板时。
- 用户把 `https://github.com/ZainDarcy/codex-team-kit-router` 或本模板 fork 地址发给 AI，并要求接入团队模板时。
- 第一次走 `Team` 路由或准备派发子代理前。
- `Docs/03-团队/Agents/团队名册.md` 状态仍为 `pending` 时。
- `.codex/agents/*.toml` 新增、删除或替换默认核心 agent 后。

## 初始化输入

初始化前，Codex 主线程读取或合并：

- `AGENTS.md`
- `Docs/README.md`
- `Docs/03-团队/开发团队.md`
- `Docs/03-团队/行业扩展包/README.md`
- `.codex/team-kit.toml`
- `.codex/agents/*.toml`
- 本文件

不要读取历史工作记录全文。

## Staging 接入协议

用户提供 GitHub 地址时，主线程不得把模板仓库直接拉取到目标项目根目录，也不得把 `codex-team-kit-router/` 作为项目子目录长期保留。

必须按以下顺序执行：

1. 新建 staging 目录拉取模板，优先放在目标项目根目录外。
2. 如果权限不允许使用根目录外路径，可临时使用项目内 `_codex-team-kit-router-staging/`，但必须写入 `.gitignore` 并在完成后删除。
3. 在 staging 目录中读取模板、探测目标项目、生成合并清单和路径映射。
4. 只把被选择的文件、配置和行业扩展 agents 合并到目标项目，不整目录覆盖。
5. 合并、初始化、体检完成后删除 staging 目录。
6. 如果 staging 无法删除，初始化不得标记为完全完成；最终汇报必须列出残留路径、原因和建议处理方式。

禁止事项：

- 不在目标项目根目录保留 `codex-team-kit-router/`、`_codex-team-kit-router-staging/` 或 `.codex-kit-staging/`。
- 不把 staging 目录加入团队名册、项目文档路由或长期文件清单。
- 不让未选择的行业扩展包从 staging 泄漏进 `.codex/agents/`。

## 探测目标项目

复制或合并模板前，主线程必须先探测目标项目，而不是直接整目录覆盖：

- 识别现有文档根目录：`docs/`、`Docs/` 或项目已有自定义目录，只能选择一个 `docs_root`。
- 识别已有 `AGENTS.md`、`AGENT.md`、项目规范、项目进度和 AI 工作流文档。
- 识别项目是否已经是 Git 仓库，以及 `.gitignore` 是否包含 `.DS_Store`。
- 扫描同类真相源是否重复，例如根目录、文档根目录和 `01-项目/` 下同时存在项目进度。
- 写入或更新 `.codex/team-kit.toml`，把 `docs_root`、项目规范、项目进度、团队目录和公共文件清单渲染成目标项目的实际路径。

如果目标项目已有稳定规范或进度文件，优先采用已有真相源；只有用户明确同意迁移时，才移动到推荐结构。不要创建内容重复的桥接副本，必要时只创建索引页指向唯一真相源。

## 初始化模式

主线程必须在初始化报告中说明采用哪种模式：

- 最小模式：只接入 Router、公共文件写锁、派发协议、核心 agents 和体检脚本；成员档案、交付模板和工作记录目录在第一次 Team 路由前再创建。
- 团队就绪模式：推荐默认。接入核心团队，并立即随机起名、创建团队名册和成员档案；不创建虚假的初始工作记录。
- 完整模式：在团队就绪模式基础上保留交付模板，适合用户明确要求长期团队沉淀的项目。

如果用户明确说“初始化团队”“允许团队流程”或要求马上使用子代理，默认使用团队就绪模式。

## 行业扩展包选择

初始化时必须显式选择是否加入行业扩展包。不得跳过，也不得默认偷偷加入。

可选项：

- `none`：不加入任何行业扩展包。
- `game-basic`：加入游戏基础小队：`game_designer`、`gameplay_engineer`、`ui_artist`、`playtest_researcher`。
- `game-full`：加入完整游戏小队：策划、程序、美术和验证分组全部加入。
- `custom`：按分组或单个 agent 选择，例如只加入战斗策划、数值策划和 UI。

如果目标项目明显是游戏项目，主线程可以推荐 `game-basic` 或 `game-full`，但初始化报告中仍必须写明最终选择。以后新增其他行业扩展包时，也必须沿用“先选择，再复制到 `.codex/agents/`，再建档”的流程。

选择 `game-basic`、`game-full` 或 `custom` 后，主线程必须：

1. 从 `.codex/agent-packs/game/` 复制被选中的 `.toml` 到 `.codex/agents/`。
2. 为新增 agent 随机生成目标项目语言下的真实人名。
3. 在 `Docs/03-团队/Agents/团队名册.md` 新增这些 agent。
4. 为新增 agent 创建成员档案。
5. 在初始化报告里说明哪些扩展包未加入，避免后续 AI 误以为遗漏。

## 起名规则

- 按目标项目主要用户的语言习惯起名。
- 名字必须像真实团队成员的人名，不要用岗位名、功能名、诗化称号或工具名。
- 中文项目使用随机的二字或三字中文人名。
- 英文项目使用自然英文人名。
- 其他语言项目使用对应语言下自然的人名。
- 每个成员名字必须唯一。
- `.codex/agents/*.toml` 的 `name` 字段保持英文机器名，不随显示名变化。
- 不使用本模板文档中的示例名作为固定输出；每个目标项目都重新随机生成。

## 必须创建或更新

团队就绪模式和完整模式必须完成：

1. 更新 `Docs/03-团队/Agents/团队名册.md`，把状态改为 `initialized`。
2. 为 `.codex/agents/*.toml` 中每个默认核心 agent 创建成员档案：

```text
Docs/03-团队/Agents/成员档案/{agent_name}-{随机显示名}.md
```

3. 每个成员档案必须基于 `_成员档案模板.md`，并写入：

- Agent 岗位
- 显示昵称
- 昵称语言
- 首次启用日期
- 适用任务
- 不适用任务
- 工作风格
- 项目理解
- 常见风险
- 适合继续派发的任务

4. 不创建初始工作记录；工作记录只在真实任务执行后创建。

最小模式必须保证第一次进入 `Team` 路由或派发子代理前，先补齐以上团队名册和成员档案。不得在团队尚未初始化时直接派发子代理。

## 初始化后检查

- `团队名册.md` 中每个默认核心 agent 都有一行。
- 每一行都有对应的成员档案文件。
- 没有重复显示名。
- 没有使用岗位名作为显示名。
- `.codex/team-kit.toml` 中的路径和实际项目路径一致，没有混用 `Docs/` 与 `docs/`。
- 同类真相源不超过一个；索引页只能指向真相源，不能复制内容。
- 初始化报告包含行业扩展包选择：`none`、`game-basic`、`game-full` 或 `custom`。
- 如果选择行业扩展包，被选中的 `.toml` 已复制到 `.codex/agents/`，并已创建名册和成员档案。
- staging 目录已经删除；如果没有删除，已明确列出残留路径和原因。
- 体检脚本只报告本地模板/项目结构问题，不当作官方 Codex 规则校验。
- 名册和档案只写摘要，不粘贴长上下文。
