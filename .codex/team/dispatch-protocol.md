# 子代理调度协议

## 总原则

- Codex 主线程是调度者、整合者和公共文件唯一修改者。
- 子代理是专业执行者，只处理主线程派发的局部任务。
- 子代理身份由 `.codex/agents/*.toml` 固化，任务上下文由主线程派发时补足。
- 子代理只接收压缩上下文包，不接收整套项目文档。
- 没有用户明确要求子代理、并行 agent 或团队流程时，不派发子代理。
- 日常 Team 与初始化/迁移分流：`Team` 只处理已初始化项目的运行态协作；初始化、迁移、扩展包选择走 `Team-Init`。
- Team-lite / delta policy 生效：每波至少一份工作记录并逐一列出参与者贡献；团队名册按参与者更新最近任务和任务次数；成员档案仅在新增可复用经验时更新。
- 只读 / dry-run 且未启动子代理、未改文件时，不落盘团队记录；最终回复说明未沉淀原因。

## Delegation Card

主线程派发前必须先写 Delegation Card。它是本波子代理的唯一任务边界。

```markdown
## Delegation Card

- 路由：
- 用户是否明确授权子代理：
- 本波目标：
- 主线程保留的关键路径：
- 可并行任务：
- 不应派发的任务：
- 预计涉及文件：
- 公共文件写锁：
- 验收方式：
- 等待策略：
- 所有本波子代理状态：
```

如果进入 `Team` 路由但没有输出 Delegation Card，主线程必须在工作记录中登记为流程偏差，并补写本波实际分工、等待状态和公共文件写入时点。

## 压缩上下文包

每个子代理只能收到与自己任务相关的压缩上下文包，内容包括：

- 任务目标
- 必要背景
- 关键文件或证据
- 允许修改路径
- 禁止修改路径
- 交付物
- 成员档案路径
- 工作记录路径
- 贡献汇报字段
- 验收标准
- 未知项和风险

不要把 `Docs/`、`.codex/team/` 或历史记录整篇发送给子代理。

## 派发前检查

主线程派发前必须写清：

- 任务目标
- 背景输入
- 主责 agent
- 协作 agents
- 验收 agent
- 允许修改路径
- 禁止修改路径
- 交付物
- 成员档案路径
- 工作记录路径
- 贡献汇报字段：身份、完成事项、产出物、验证、建议回写
- 验收标准

## 波次执行

- Wave 0：主线程理解需求、选择路由、按需读取文档、写 Delegation Card。
- Wave 1：探索/研究/方案代理并行，只给证据和建议。
- Wave 2：实现代理并行，写入范围必须互不重叠。
- Wave 3：QA/reviewer 验收。
- Wave 4：主线程或 `docs_keeper` 按 Team-lite / delta policy 更新团队沉淀（至少一份工作记录，团队名册按参与者行级更新，成员档案按新增经验更新）。
- Wave 4.5：主线程确认所有本波子代理状态为 completed/closed，再统一更新公共文件、进度和团队记忆。
- Wave 5：主线程汇总团队贡献，向用户说明谁干了什么、产出了什么、验证了什么。

## 等待与关闭

- 主线程启动一波子代理后，等待该波全部完成。
- 不在子代理运行中途随意改任务或打断。
- 子代理完全完成后，主线程主动关闭其线程，释放 `max_threads` 席位。
- 后续专家代理在下一波启动。
- 公共文件写入前，工作记录或主线程摘要必须写明所有本波子代理状态；紧急例外必须写清原因和风险。

## 输出要求

子代理最终输出必须包含：

- 做了什么
- 改了哪些文件
- 验证了什么
- 没验证什么
- 风险
- 成员档案路径
- 工作记录路径
- 成员档案更新草稿
- 工作记录草稿
- 建议回写到公共文件的内容

主线程最终输出必须额外包含：

- 团队贡献汇报
- 每个参与者的产出
- 团队名册、成员档案和工作记录是否已落盘
- 被采纳的建议
- 未采纳建议及原因
- 未启用子代理时的原因

## 禁止事项

- 不把主线程下一步立即需要的关键路径交给子代理。
- 不让多个写入型子代理修改同一批文件。
- 不让子代理修改公共文件。
- 不让子代理启动下一层子代理。
- 不把探索日志、长命令输出或无关文档原文回灌到主线程。
- 团队贡献汇报只做压缩摘要，不回灌子代理完整工作记录。
- 不把“写工作记录”当成可选项；read-only 子代理无法写文件时，必须由主线程或 `docs_keeper` 代写。
- 不把初始化文档当成日常 Team 的默认读取项；只有 `Team-Init` 路由才读取初始化/迁移文档。
